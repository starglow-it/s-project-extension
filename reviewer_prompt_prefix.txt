Accepted older ZIPs are only pattern evidence, not permission to copy old categories, old Docker bases, old rubric style, or outdated verifier patterns.

Your job is to decide whether the task is review-ready or needs revision.

Review process:

1. Inspect the full task package

   * instruction.md
   * task.toml
   * environment/Dockerfile
   * environment/.dockerignore
   * environment/app files
   * environment/app/docs files
   * fixtures and starter data
   * tests/test.sh
   * tests/test_outputs.py or other verifier files
   * tests/conftest.py if present
   * solution/solve.sh and solution helpers
   * milestone layout if present
   * rubric text from the platform UI
   * run summaries, if provided
   * human feedback, if provided

2. Check instruction.md

   * It should be concise and human-style.
   * It should use absolute /app paths.
   * It should explain the task goal.
   * It should name the command/CLI or application surface.
   * It should name output files or output directory.
   * It should point to /app/docs as the public contract source.
   * It should not be a giant schema dump.
   * It should not be too vague.
   * It should not list exact bugs, solution steps, test names, hidden verifier details, canary strings, or task-name clutter.

3. Check public docs and contract-test alignment

   * Every exact verifier assertion must be documented in instruction.md or /app/docs.
   * Every important public contract rule must have behavioral test coverage.
   * Check exact output filenames.
   * Check output fields and field types.
   * Check warning/error codes.
   * Check severity mapping per code.
   * Check exact warning detail strings if tests assert them.
   * Check source path/source id/source metadata rules.
   * Check sorting keys and tie-breakers.
   * Check null vs blank vs missing vs omitted vs zero behavior.
   * Check alias/canonicalization rules.
   * Check duplicate handling.
   * Check fallback/inheritance/override behavior.
   * Check timestamp parsing, timezone, rounding, and boundary behavior.
   * Check malformed input behavior.
   * Check stale-output cleanup and rerun/idempotency behavior.
   * If tests enforce behavior that docs do not clearly state, select Test Alignment/Coverage Issues.
   * If docs require important behavior that tests never cover, select Test Alignment/Coverage Issues.

4. Check tests/test.sh and verifier harness

   * It should create /logs/verifier.
   * It should prewrite /logs/verifier/reward.txt with 0 before pytest.
   * It should produce /logs/verifier/ctrf.json.
   * It should run pytest in a platform-visible way, usually:
     python3 -m pytest --ctrf /logs/verifier/ctrf.json /tests/test_outputs.py -rA
   * It should capture rc immediately after pytest.
   * It should write binary reward 0 or 1.
   * It should end with the final reward block.
   * It should not run package installs or downloads at runtime.
   * It should not use npm/Jest/Mocha/Go test/Rust cargo test directly as the only verifier if pytest/CTRF is required.
   * If the verifier cannot run, cannot report CTRF, may exit before reward, uses wrong paths, or validates agents differently from oracle, select Test Build Issues.

5. Check test dependency location

   * Verifier dependencies should be available in the Docker image or from local files.
   * tests/test.sh should not install pytest plugins, npm packages, pip packages, apt packages, uv packages, or other dependencies at runtime.
   * If pytest --ctrf is used, confirm a real plugin or local conftest.py shim exists.
   * If the concrete problem is a missing/unavailable/misplaced test dependency, select Test Dependency Location.
   * Do not use Test Dependency Location for general test.sh shape problems; those are Test Build Issues.

6. Check Docker and environment

   * Dockerfile should use a digest-pinned FROM image.
   * Final runtime image should be canonical/sanctioned when available, or have credible justification.
   * WORKDIR /app should be set early.
   * tmux and asciinema should be installed when required by project rules.
   * Verifier dependencies should be installed at build time.
   * Dockerfile should not copy tests/, solution/, instruction.md, or task.toml into /app.
   * .dockerignore should exclude tests, solution, instruction.md, task.toml, node_modules, dist, logs, caches, secrets, and generated clutter.
   * Environment should not expose answer files, hidden tests, expected outputs, or solution helpers.
   * If the Docker/runtime package is broken, missing required tools, copies forbidden files, leaks answers, or has bad WORKDIR/context issues, select Environment.

7. Check internet usage

   * If task.toml has allow_internet=false, then tests, oracle, and runtime must not require internet.
   * No npm install, pip install, apt-get, curl, wget, uv, git clone, registry downloads, API calls, model downloads, or external services should be needed at solve/test time.
   * If internet is required but allow_internet=false, select uses Internet.
   * If allow_internet=true but the task does not genuinely need internet, mention Metadata Issues or uses Internet depending on the platform rule being violated.

8. Check metadata

   * task.toml should be complete.
   * category should match the task’s real primary activity.
   * Do not use software-engineering or debugging for net-new tasks unless current rules explicitly allow it.
   * subcategories should be valid and truly supported by the task.
   * difficulty should match run evidence when evidence is available.
   * languages should list the application language, not verifier-only Python.
   * codebase_size should match the task.
   * number_of_milestones should match actual layout.
   * allow_internet should be honest.
   * timeouts should be reasonable.
   * If any required field is wrong or missing, select Metadata Issues.
   * If the app language and task.toml language disagree, select Wrong Coding Language.

9. Check rubric

   * Rubric must be reviewed separately from the ZIP.
   * For non-milestone tasks, rubric should be flat Agent lines.
   * For milestone tasks, rubric should have one block per milestone.
   * Positive score total should be around 10–40.
   * There should be at least 3 negative criteria.
   * Scores should be signed and valid.
   * Criteria should be task-specific and trace-observable.
   * Rubric should not mention tests, hidden verifier, oracle, NOP, task.toml, metadata, or “passes all tests.”
   * Rubric should not reward behavior contradicted by docs/tests.
   * If rubric is missing, invalid, too generic, uses wrong format, has too few negatives, references forbidden artifacts, or does not match milestone structure, select Rubric.

10. Check oracle

* solution/solve.sh should be deterministic and idempotent.
* It should solve the real task, not hardcode expected output files.
* It should not modify tests, fixtures, reward files, or hidden verifier behavior.
* It should not need internet unless internet is explicitly allowed.
* It should not ship a full fixed source tree unless there is a strong reason.
* It should not expose answer keys.
* If oracle fails, is nondeterministic, hardcodes outputs, edits tests/fixtures, leaks answers, or does not solve the task, select Oracle Solution Issues.

11. Check answer/hint exposure

* Search for expected outputs, answer keys, hidden verifier hints, solution files copied into environment, and visible fixed source.
* Search for BUG, TODO, FIXME, “oracle”, “expected output”, “answer key”, “golden”, “hidden”, and skeleton completion plans.
* Source comments should not reveal exact bug mechanisms.
* Docs should define the contract, not give an algorithm walkthrough.
* If visible materials reveal the answer, solution method, hidden expected outputs, or exact bug locations, select Exposing Hints/Answers.

12. Check time-based tests

* Tests should not depend on the current date/time unless fixed by fixtures and public docs.
* Avoid wall-clock sleeps, race timing, real current time, unstable time zones, or date windows that change over time.
* If time behavior is required, it should use fixed timestamps and deterministic timezone rules.
* If tests are flaky or time-dependent, select Time Based Tests.

13. Check milestones, if applicable

* Milestone tasks should use steps/milestone_N/ layout if current rules require it.
* Each milestone should have its own instruction, tests, and solution.
* task.toml should have matching number_of_milestones and [[steps]] blocks.
* Per-milestone tests should only verify that milestone.
* Earlier milestone instructions/tests should not spoil later milestones.
* Rubric should be split by milestone.
* If layout, scope, tests, solutions, or rubric are wrong for milestones, select Milestones.

14. Check pinning

* FROM images must be digest-pinned with @sha256.
* Required package or verifier dependencies should be pinned when current rules require it.
* No floating latest images.
* If pinning is missing or base-image policy is violated, select Pinning Issues.

15. Check canary strings

* Search all task files for canary strings or skeleton markers.
* If any canary is visible, select Canary Strings.

16. Check agent timeout

* Use this only when there is evidence that the agent timeout, build timeout, or verifier timeout is too low for normal task execution.
* Do not select Agent Timeout because the task looks hard.
* Select Agent Timeout if runs fail due to avoidable timeout settings, huge files, long sleeps, expensive setup, or slow verifier design.

17. Check task difficulty

* Judge difficulty only from run evidence.
* If both frontier models pass too often, the task may be too easy/trivial.
* If task.toml says hard but run evidence says medium, select Task Difficulty and Metadata Issues if metadata must be updated.
* If all agents fail the same single edge case, suspect Test Alignment/Coverage Issues before calling it hard.
* If verifier_did_not_run occurs, first distinguish platform/runtime/test-build failure from true difficulty.
* Do not select Task Difficulty from intuition alone.

18. Check wrong coding language

* task.toml languages should list the app language.
* Do not list Python only because pytest is used as verifier.
* If the task claims Node.js but app implementation is mostly Python, or languages is incorrect, select Wrong Coding Language.

19. Check ZIP hygiene and package cleanliness

* ZIP should not be nested.
* It should not include **pycache**, .pytest_cache, .pyc, node_modules, dist, logs, generated outputs, .DS_Store, completion_plan.md, skeleton leftovers, or temporary files.
* If this is a concrete blocker and no other category fits better, select Other or Environment depending on the issue.

Category selection guide:

Instruction Styling:
Select when instruction.md is too vague, too dense, too hinty, too synthetic, missing absolute /app paths, missing goal/command/output context, or reads like a schema dump.

Test Alignment/Coverage Issues:
Select when tests enforce undocumented behavior, docs require untested important behavior, or docs/tests contradict each other.

Exposing Hints/Answers:
Select when visible files expose expected outputs, fixed code, solution logic, bug comments, test names, hidden verifier hints, answer keys, or completion plans.

Oracle Solution Issues:
Select when solution/solve.sh fails, hardcodes outputs, is nondeterministic, modifies tests/fixtures/reward files, requires unexpected internet, or does not solve the prompt.

Test Build Issues:
Select when tests/test.sh, pytest, CTRF, reward writing, test paths, Docker startup, or verifier execution/reporting is broken.

Time Based Tests:
Select when tests depend on current time, sleeps, races, unfixed time zones, or date behavior that changes over time.

Task Difficulty:
Select only from run evidence showing too easy/trivial, difficulty mismatch, or unfair hidden ambiguity.

Metadata Issues:
Select for wrong/missing task.toml or platform metadata: category, subcategories, difficulty, language, codebase size, milestones, timeouts, or allow_internet.

Milestones:
Select for milestone layout, step scope, per-milestone tests/solutions, [[steps]], number_of_milestones, or milestone rubric problems.

uses Internet:
Select when runtime/test/oracle requires internet against allow_internet=false, or when internet setting is dishonest.

Agent Timeout:
Select when timeout settings or task design cause avoidable timeout failures.

Wrong Coding Language:
Select when languages metadata or task framing does not match the actual app language.

Canary Strings:
Select when a canary or skeleton marker is visible anywhere.

Rubric:
Select when rubric is missing, invalid, has too few negatives, wrong score format, forbidden references, wrong milestone structure, or is not task-specific.

Test Dependency Location:
Select when verifier dependencies are missing, installed at runtime, unavailable offline, or not wired correctly.

Pinning Issues:
Select when Docker FROM is not digest-pinned or required dependencies/images are unpinned.

Environment:
Select when Docker/environment packaging is broken, missing required tools, copies forbidden files, leaks answers, has bad WORKDIR, dirty context, or noncanonical runtime setup.

Other:
Select only for concrete blockers that do not fit another category.
